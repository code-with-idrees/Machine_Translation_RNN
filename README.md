# 🌐 English ➔ Urdu Neural Machine Translation 

A complete from-scratch implementation of an **English to Urdu Neural Machine Translation (NMT)** model using a **Vanilla RNN Encoder-Decoder architecture** in PyTorch. 

This repository was designed specifically for **Generative AI Assignment #1** and adheres strictly to the constraints and rubrics provided—implementing the entire pipeline without the use of advanced gating mechanisms (LSTMs/GRUs) or Attention.

## 🧠 Model Architecture (Vanilla RNN Encoder-Decoder)

The model consists of two primary components operating identically on simple `torch.nn.RNN` layers:

1.  **Encoder:** Reads the English sentence token by token, updating a hidden state. The final hidden state $h_T$ acts as the **Context Vector**, compressing the entire source meaning.
2.  **Decoder:** Initialised with the context vector, it generates the Urdu sentence one token at a time. It employs **Teacher Forcing** during training to stabilize learning.

## 📖 Walkthrough & Pipeline Documentation

The codebase is structured into **12 perfectly modular sections**. Below is a breakdown of the pipeline addressing the assignment rubric.

### 1️⃣ Environment & Setup
All random seeds (PyTorch, NumPy, Python) are locked (`SEED=42`) to guarantee **deterministic reproducibility**. The device dynamically maps to GPU (CUDA) if available, falling back to CPU.

### 2️⃣ Data Loading & Exploration (`dataset_statistics.ipynb` & Section 2)
The dataset is comprehensively analyzed. We check for shape (`9103 pairs`), inspect data types, memory footprints, missing values, and precise corpus lengths. 

### 3️⃣ Dataset Preprocessing
We implemented distinct cleaning pipelines:
*   **English:** Lowercase, Unicode NFKC normalization, URL/email stripping, regex-based punctuation clearing.
*   **Urdu:** Zero-width character removal, custom Urdu regex mapping, Urdu punctuation normalisation.
*   **Quality Filtering:** Pairs where one side is empty, or the length ratios are extreme (e.g., `< 0.3` or `> 3.0`) are meticulously filtered out.

### 4️⃣ Train/Validation/Test Split
A robust **80/10/10 Split** is implemented. 
*   **Crucial implementation:** We grouped by *unique English source sentences* and performed a stratified split. This explicitly **guarantees zero data leakage**; no English sentence from the Test or Val sets appears in the Train set.

### 5️⃣ Tokenisation & Vocabulary Construction
A custom `Vocabulary` class dynamically builds from the training set only. 
*   **Special Tokens:** `<pad>`, `<bos>`, `<eos>`, `<unk>`.
*   **Min Frequency Filter:** Words appearing fewer than 2 times are mapped to `<unk>`. This dramatically trims the vocabulary tail, improving learning efficiency within memory limits.
*   Final source and target vocabulary sizes are tracked.

### 6️⃣ Sequence Encoding, Padding & Batching
*   **NMTDataset:** Wraps the tokenized inputs into PyTorch `Dataset`.
*   **Dynamic Padding (`collate_fn`):** Instead of padding every sentence to a global max length, we dynamically pad sequences *to the maximum length of the current batch*. This heavily reduces wasted computation.
*   Batch shapes and tensors are validated before training.

### 7️⃣ Vanilla RNN Decoder Implementation
Strictly adhered to the assignment constraints. We utilize `nn.Embedding`, `nn.RNN`, and a final `nn.Linear` projection. We also implemented **Label Smoothing Loss** ($ε = 0.1$) to prevent overconfidence.

### 8️⃣ Model Training
The training loop utilizes:
*   **Adam Optimizer** with specific betas `(0.9, 0.98)`
*   **Gradient Clipping** (`max_norm = 1.0`): Absolutely crucial for Vanilla RNNs to prevent exploding gradients.
*   **ReduceLROnPlateau Scheduler:** Halves learning rate if validation loss stagnates.
*   **Early Stopping:** Halts training safely after 5 epochs of no improvement.

### 9️⃣ Hyperparameter Tuning (Grid Search)
An automated Grid Search tests 8 combinations covering:
*   `embed_dim`: 128, 256
*   `hidden_dim`: 256, 512
*   `n_layers`: 1, 2
*   `lr`: 1e-3, 5e-4
*   `dropout`: 0.2, 0.3
*   `batch_size`: 32, 64

The best configuration based on validation loss is automatically selected and retrained for the full duration.

### 🔟 Inference, Decoding & Evaluation
Two inference methods are compared:
*   **Greedy Decoding:** Fast, taking `argmax` at every step.
*   **Beam Search Decoding ($k=4$):** Maintains the top 4 hypothesis simultaneously using length-penalized log-likelihoods.
*   **BLEU Score:** We report Corpus `BLEU-1` through `BLEU-4`. 

### 1️⃣1️⃣ Error Analysis & Research Discussion
We pass translated outputs through a 9-category **Error Taxonomy**:
1. Repetition Loops
2. Complete Hallucinations
3. Severe Under-generation
4. Severe Over-generation
5. OOV Cascades
6. Poor Reordering
7. Partial Matches
8. Near Misses
9. Acceptable Translations

#### 🚨 Out-of-Distribution (OOD) Analysis
We specifically test the model against sentences in the 90th percentile of length, or sentences containing $>2$ OOV tokens, explicitly measuring the BLEU drop from In-Distribution to Out-Of-Distribution.

#### ❌ Limitations of Vanilla RNNs
As requested in the rubric:
1.  **Vanishing Gradients:** `tanh` activations cause gradients to shrink over long sequences.
2.  **Information Bottleneck:** The *entire* source sentence is bottlenecked into a single hidden vector ($h_T$). 
3.  **Poor Reordering Capacity:** English (SVO) $\to$ Urdu (SOV) requires significant long-range mapping that simple RNNs struggle with.
4.  **Repetition Traps:** Without attention to track "coverage", the decoder gets stuck generating the same tokens.
